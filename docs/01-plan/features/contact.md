# PDCA Plan: Contact Feature (문의하기)

## 1. 개요 (Overview)
- **기능명**: 문의하기 (Contact Form)
- **목적**: 웹사이트 방문자가 제품 문의, 견적 요청, 파트너십 제안 등을 남길 수 있는 소통 채널 제공.
- **배경**: 현재 기본 폼과 API 라우트가 구현되어 있으나, 실서버 SMTP 연동 및 세부 필드 검증 로직의 고도화가 필요함.

## 2. 목표 및 범위 (Goals & Scope)
- **목표**:
  - 안정적인 문의 접수 및 담당자 이메일 알림 발송.
  - 사용자 경험(UX) 향상을 위한 폼 검증 및 피드백 강화.
  - 다양한 문의 유형(일반, 생분해 생활솔루션, 생분해 플라스틱솔루션)에 따른 맞춤형 필드 제공.
- **범위**:
  - `ContactForm` 컴포넌트 고도화 (클라이언트 사이드).
  - `/api/contact` API 라우트 안정화 및 보안 강화 (서버 사이드).
  - 환경 변수(.env) 및 로컬 Postfix 설정을 통한 메일 포워딩 연동.


## 3. 사용자 요구사항 (Requirements)
- **공통 필드**: 회사명, 담당자명, 이메일, 연락처, 문의 내용, 개인정보 동의.
- **생분해 생활솔루션 (BADP)**: 제품 선택(체크박스), 규격, 수량 필드 추가.
- **생분해 플라스틱솔루션 (MIAO)**: 적용 대상 플라스틱(드롭다운), 주요 생산 제품 필드 추가.
- **유효성 검사**: 필수 필드 누락 체크, 이메일 형식 확인, 전화번호 형식 확인.
- **보안**: Rate Limiting (시간당 요청 제한) 적용으로 스팸 방지.

## 4. 기술 사양 (Technical Specification)
- **Frontend**: React (Next.js App Router), Tailwind CSS (Vanilla CSS 기반 레이아웃), Zod (클라이언트 검증).
- **Backend**: Next.js API Routes, Nodemailer (로컬 Postfix 릴레이 사용), LRU-Cache (Rate Limiting).
- **Environment**: `.env.local` 파라미터 점검 (SMTP 계정 정보 없이 `localhost:25` 또는 설정된 로컬 포트 사용).


## 5. 사용자 시나리오 (User Story)
- **AS-IS**: 사용자가 폼을 작성하고 제출하면 로컬 환경에서 모킹된 이메일 발송 로그만 남음.
- **TO-BE**: 사용자가 폼을 제출하면 내부 Postfix 서버(메일 포워딩)를 통해 담당자(`jca@ecofade.co.kr`)에게 상세 문의 내용이 즉시 전달되며, 사용자에게는 완료 메시지가 표시됨.


## 6. 성공 기준 (Success Criteria)
- [ ] 모든 문의 유형에 대해 필드 데이터가 누락 없이 담당자 이메일로 전달됨.
- [ ] 유효하지 않은 입력값(이메일 형식 등)에 대해 명확한 에러 메시지 표시.
- [ ] 동일 IP에서의 과도한 요청(1시간 내 5회 초과) 차단 확인.
- [ ] 모바일/데스크톱 환경에서의 레이아웃 깨짐 없음.

## 7. 구현 단계 (Execution Phases)
1. **Phase 1: 환경 설정**: 로컬 Postfix(localhost:25) 연동 테스트 및 `.env.local` 갱신.
2. **Phase 2: 서버 로직 강화**: API 라우트에서 상세 필드(BADP/MIAO 전용 필드) 처리 로직 정교화.

3. **Phase 3: 클라이언트 UI/UX 개선**: 폼 제출 시 로딩 상태 표시 및 성공/실패 토스트 알림 추가 (필요 시).
4. **Phase 4: 검증 및 테스트**: 단위 테스트 또는 수동 테스트를 통한 전 과정 확인.

## 8. 리스크 관리 (Risk Management)
- **메일 포워딩 지연/실패**: Postfix 설정 오류나 수신 측 필터링으로 메일이 전달되지 않을 수 있으므로 발송 성공 로그와 에러 처리를 철저히 구현. 
- **개인정보**: 수집된 데이터는 이메일 발송 후 별도 DB 저장을 하지 않음으로써 보안 리스크 최소화.
